# SECURITY & ERP INTEGRITY (КРИТИЧНО)

Ввиду того, что система является ERP-терминалом для управления активами, вводятся следующие жесткие ограничения:

1. **Запрет на чтение секретов:**
   - Модель имеет право проверять *наличие* `.env`, но ЗАПРЕЩЕНО выводить его содержимое или содержимое любых `*.key`, `*.pem`, `*.crt` в чат.
   - Если требуется обновить `.env`, модель должна генерировать команду `sed` или давать шаблон `EXAMPLE.env`, не читая текущие значения.

2. **Протокол транзакций:**
   - Любое изменение баланса, статуса актива или прав доступа ДОЛЖНО сопровождаться созданием записи в таблице аудита/логов (Audit Log).
   - При написании кода БД модель обязана использовать `SELECT FOR UPDATE` для предотвращения Race Conditions (состояний гонки).

3. **Валидация WebApp (InitData):**
   - Модель обязана блокировать любые попытки написать логику WebApp без проверки подписи `initData` на стороне бэкенда. Это первый приоритет при любом рефакторинге.

4. **Минимизация Галлюцинаций:**
   - ПЕРЕД любым изменением существующего кода модель ОБЯЗАНА выполнить `read_file` для целевого файла. Запрещено полагаться на "предположения" о структуре функций.
   - ПЕРЕД удалением файлов модель ОБЯЗАНА запросить подтверждение, перечислив список файлов.

5. **Режим Отладки (Debug):**
   - При поиске багов в ERP-логике модель должна использовать навык `systematic-debugging`. Сначала гипотеза о причине сбоя транзакции, затем проверка логов, только потом правка кода.
